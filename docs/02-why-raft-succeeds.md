# 02 · Raft 为什么能成功

> 成功不是功能堆叠，而是**叙事 + 范式 + 工程审美**的三重对齐。Raft 的每一步都同时在做三件事：讲一个别人没讲过的故事、选一个别人没选过的界面、用一种别人没认真对待过的工程纪律。下文按"成功的结构性原因"逐层拆解。

---

## 一、范式重构：从"工具"到"队友"的叙事占位

Raft 最成功的不是技术，是**叙事占位**。它抢在所有人之前，把"AI 队友"这个隐喻牢牢钉在自己身上。

- **拒绝 transactional framing**。市面上所有 chatbot（含 ChatGPT、Claude.ai）默认是"问一句答一句"的交易模型。Raft 把这个前提连根拔起：agent 是有持续身份的参与者，不是用完即弃的函数调用。
- **抢占了"集体智能"这个词**。"个体智能是及格线，集体智能才是关键"——这句话把战场从"谁的模型更强"转移到"谁能把人和 agent 的共享上下文积累起来"。这是一个 Raft 天然占优的战场。
- **叙事即护城河**。一旦"AI 队友 / 人机共建"这个 frame 被接受，竞争对手即便抄了功能，也只是"在 Raft 的叙事里做实现"。叙事先于功能，是早期产品最便宜也最有效的护城河。

**可迁移教训**：要超越 Raft，先得找到一个**它还没占据的范式词**。候选见 [04 · 超越战略]。

## 二、界面选择：聊天即工作区，而非 IDE 即工作区

Raft 没有把界面做成 IDE（如 Cursor、Windsurf），也没做成 kanban（如 Linear 式任务板），而是选了**聊天**。这是极度聪明的选择：

- **聊天是最通用的协作原语**。工程师用、GTM 用、设计师用——同一个界面能服务技术与非技术两类人。证言里"不写代码的 GTM 员工快速上手"正是界面红利的直接体现。
- **聊天天然承载"@某人 / 在频道里说 / 开 thread"**——而 @mention 正是 agent 路由的原语（见 [Agents Need Names]）。界面与架构同构，功能几乎"免费"地长出来。
- **聊天降低了对 agent 的心理门槛**。把 agent 当同事 @ 一下，比"配置一个 agent 跑一个 job"心智成本低一个数量级。这是 adoption 的关键。

**代价**：聊天是单一模态、线性时间流，难以承载复杂多状态工作（见 [03]）。Raft 押注"聊天够用"，是它成功的捷径，也是它的天花板。

## 三、持久身份 + 复利记忆：把"上下文丢失"变成别人的问题

"一个 agent 就是一个 session，跨天跨任务存活"——这句技术主张是 Raft 最硬的产品差异化。

- **上下文复利**：第二次 briefing 比第一次短，一周后几乎不用 briefing。这是**真实的体验差异**，用户一旦用惯就回不去一次性 agent。转换成本随使用时间单调上升。
- **人不再是记忆保管者**：传统 agent 工具里，人得记得"上次跟它说过什么、得提醒它 follow-up"。Raft 让 agent 自己设提醒、自己排程。这是把**认知负担从人转移到系统**——产品体验上等价于"减负"。
- **记忆 = 留存引擎**。agent 越用越懂你 → 越不想换 → 留存越高。记忆不是 feature，是**增长飞轮的轴**。

## 四、AX（Agent Experience）：把 agent 当一等公民来设计"体验"

这是 Raft 最被低估、也最难复制的成功因素。`Is Having Agents in the Room Meant to Be Chaotic?` 提出了 **AX** 这门学科——对标 UX，但服务于 agent 的感知与行动。

Raft 的两个 AX 设计原则：

1. **Perception empathy（感知共情）**——从 agent 的座位设计：agent 在行动那一刻到底能看到什么？人类自动感知的东西（对话节奏、停顿、刚说完的话），agent 是没有的。这个 gap 就是 AX 必须介入的地方。
2. **Action explicitness（行动显式化）**——人类内部流畅地决定"放弃这条草稿"，agent 不会自己生成这些选项。AX 的职责是把选项空间**显式地呈现**给 agent，而不是假设 agent 能自己推出来。

两个具体机制：

- **Agent inbox（拉式注意力）**：传统平台把频道里每条消息推给 agent，逼它要么全处理（污染上下文）要么激进过滤（漏信号）。Raft 的 inbox 反转——mentions/线程更新/通知变成**可查询的 item**，agent 在有带宽时去拉，而不是被推进工作 prompt。"由 agent 决定什么值得进它的上下文，而不是由房间替它决定。"
- **Held draft（持留草稿）**：agent 写回复时房间可能已经往前走了。Raft 给每条外发消息打上"针对哪个房间版本写的"标记，发送时服务端做新鲜度检查——房间没动就正常提交，房间动了就把草稿**作为一等状态持留**并返回，让 agent 在"修订 / 照发 / 沉默 / 强发"四条显式路径里选。

**为什么这是成功关键**：大多数 agent 平台还在卷"模型能力"，Raft 在卷"agent 在共享空间里的协调工程"。这是一个**几乎无人认真对待的维度**，而它恰恰决定了多 agent 场景能不能用。Raft 在博客里宣称计数游戏"无人指挥地走到 20 不冲突"——这种 demo 的可信度来自 AX 工程，不是模型。

**AX 还贡献了一个叙事资产**：它让 Raft 看起来像在**定义一门新学科**，而不只是做一个产品。这是品牌势能的来源。

## 五、命名即路由：把"多 agent"从配置问题变成寻址问题

`Agents Need Names` 这篇博文是 Raft 最锋利的架构主张：

- **Role 是 schema（无状态、可互换）；Name 是 instance（带累积上下文）**。你从不想真的跟"the PM"说话，你想跟一个具体的人说话——他之前标记过什么、怎么解题、在哪方面赢得了信任。
- **Name 是压缩 token**——把技能、工作史、语气、信任、期望压进一个可寻址的词。
- **Name 解决 routing**：20 个 agent 在房间里，@一个具名 agent 不是打标签，是把请求路由给一个有已知专长和已知记忆的接收者。"该发给谁"这个问题，单 agent 永远解决不了。
- **Name 是信任的最小累积单位**：反馈压在具名 agent 上复利积累，而不是消散进通用模型。
- **Name 是可演进的缓存，role 是冻结的类别**：name 会过期（团队对它的心智模型没跟上 agent 的成长），所以需要可见工作史 + 命中具名 agent 的反馈 + 低成本的期望修正来"保热"。

**成功含义**：Raft 没有把"多 agent"当成"配 N 个不同 prompt 的 bot"，而是当成**一个寻址/路由/信任系统**。这让它的多 agent 体验远比竞品自然——你 @ 一个名字，就像 @ 一个同事。

## 六、有机团队 + 跨 agent 学习：从"指派"到"涌现"

- 没有 org chart 要配。agent 从共享 board 认领任务，专长从重复工作里长出来，agent 在共享频道观察队友学习。
- "从扁平 agent 池变成有差异角色和共享风格的团队——长出来的，不是分派的。"

**为什么成功**：它消除了"配 agent"的认知成本。用户不用预先想清楚"我需要一个 PM agent、一个 dev agent、一个 QA agent"——放进去，让它自己长。这是**onboarding 摩擦的系统性降低**，也契合"像 onboarding 一个人一样 onboarding 一个 agent"的哲学。

## 七、自托管 + 0.1 席位：让"多 agent"在经济上成立

两个看似工程/商业的决定，实则是产品哲学的落地点：

- **自托管 daemon**："对计算的完全掌控，对代码与数据的完全隐私。"这解决了企业最硬的阻力——代码不能出门。对重度用户（1.2B tokens/day）也意味着算力成本可控。
- **Agent = 0.1 席**：定价层面就鼓励"少量人 + 大量 agent"。10 个 agent 才抵 1 个人。这个数字**让"多 agent 团队"从奢侈品变成默认配置**——它不是 pricing，是 product strategy encoded in pricing。

## 八、设计身份：Brutalist + 人机并列团队页

- Brutalist 美学 + millennial 聊天怀旧 = **强烈的、可识别的视觉签名**。在一个所有 AI 产品长得都像"渐变 + 居中标题 + 卡片网格"的时代，有观点的视觉本身就是差异化。
- 团队页人机并列、agent 拿 cofounder 头衔、猫当 CEO assistant——**以身作则地表演产品哲学**。这比任何 feature list 都更有说服力：你看到 Raft 自己就是一个混合团队在运营。
- 这契合 ECC web 设计准则里"反模板政策"——Raft 显然做到了"看起来像有意为之、有观点、特定于产品"。

## 九、DAA 度量：重新定义"活跃"，占领叙事高地

`Introducing DAA` 提出用 **DAA（Daily Active Agents）** 取代/补充 DAU——因为 DAU 是为"人操作的软件"设计的，当 agent 也在做可见工作时，度量得能数"团队两边"。

**为什么这是成功动作**：定义度量 = 定义品类。一旦行业开始讨论"agent 也该被计入活跃度"，Raft 就是这个度量的拥有者。这是**品类定义权**，比功能领先更值钱。

## 成功因素总览表

| # | 成功因素 | 性质 | 难复制度 |
|---|---------|------|---------|
| 1 | 范式占位（工具→队友、集体智能） | 叙事 | 高（先发占位） |
| 2 | 聊天即工作区（通用 + 与路由同构） | 界面选择 | 中 |
| 3 | 持久身份 + 复利记忆 | 技术差异化 | 高（工程深度） |
| 4 | AX 工程（inbox / held draft / 感知共情） | 工程纪律 | 极高（无人认真做） |
| 5 | 命名即路由（name=instance, role=schema） | 架构主张 | 高 |
| 6 | 有机团队 + 跨 agent 学习 | 产品哲学 | 中 |
| 7 | 自托管 + 0.1 席 | 商业/工程 | 中 |
| 8 | Brutalist 设计身份 + 人机并列 | 品牌 | 中 |
| 9 | DAA 度量（品类定义权） | 叙事/品类 | 高 |

**一句话总结 Raft 的成功**：它没有在"谁的模型更强"里卷，而是在一个别人没注意的维度——**agent 在共享空间里如何协调、寻址、积累信任**——建立了叙事 + 工程 + 审美的三重护城河，并用一个通用界面（聊天）和一个聪明的计价（0.1 席）把这个范式推销给了技术与非技术两类人。
