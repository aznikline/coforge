# 07 · 开源欠缺部分的实现时间估算

> 问题：开源欠缺的部分（即 [06] 的 C3 + 缝合成品），实现需要多久？
>
> **结论先行**（基线 = 3-5 人小团队 + 复用开源）：
> - 能 demo 的 **PoC**：**3-4 个月**
> - 能用、自托管、小团队真用上（≈ 追平 Raft 底座）：**6-9 个月**
> - 功能上超过 Raft（含差异化层）：**12-18 个月**
> - 企业级 + 生态（完全超越）：**18-24 个月**
>
> 全自研（不复用开源）每个数字 ×1.5-2。下面是拆解。

---

## 一、欠缺口拆成可估算模块

[06] 已确认开源欠缺的不只是"一个东西"，而是从底座到差异化的若干层。按工程难度排序：

| 模块 | 性质 | 难度 | 复用开源？ |
|------|------|------|-----------|
| M1 共享频道 UI 改造（多人+多 agent） | UI + 后端语义重写 | 中 | LibreChat fork |
| M2 Agent 持久身份接入 | 集成 | 低-中 | Letta / LangGraph |
| M3 基础 inbox（拉式注意力） | 后端 | 中 | 部分可借消息队列 |
| M4 命名路由 + 能力寻址 | 注册表 + 解析 | 中 | 无 |
| M5 Held draft 新鲜度协议 | 协议 + 服务端 | 中 | 无（Raft 私有） |
| M6 并发原语（lock/CAS/watch/presence） | 分布式正确性 | **高** | 无 |
| M7 事件流房间模型（替 turn-based 快照） | 架构重写 | **极高** | 无 |
| M8 记忆可检视治理（diff/provenance/branch/ACL） | 差异化层 | 中-高 | 无（Raft 也没做） |
| M9 分级自治 trust tier | 工程 + 产品 | 中 | 无 |
| M10 多视图 Work Graph | 数据模型 + 多 UI | 中-高 | 无 |
| M11 Enterprise（SSO/RBAC/审计/air-gap） | 工程 | 中 | LibreChat 现成 |
| M12 AX 开放协议 + SDK | 协议设计 + 生态 | 高 | 无 |
| M13 Daemon 本地 agent runner | 工程 | 中 | 部分借 OpenHands |
| M14 第一个 vertical 接入（OpenHands SWE） | 集成 | 中 | OpenHands |

**关键观察**：M6 + M7 是难度天花板（分布式系统正确性问题），M12 是设计不确定性最高的（AX 无标准答案）。这三块决定总工期。

## 二、分层时间估算（人月 + 自然月）

> 基线：3-5 人持续投入，复用开源，LLM API 现成（不自训模型）。
> 人月 = 1 人 1 个月；自然月 ≈ 人月 ÷ 团队规模，但受关键路径串行限制。

### Phase 0 — PoC：能 demo 的协作空间（追平 Raft 底座的"形"）

| 模块 | 人月 |
|------|------|
| M1 共享频道改造（LibreChat fork，多参与者 + @mention） | 2-3 |
| M2 持久身份接入（Letta 做 memory backend） | 1-2 |
| M3 基础 inbox（简化拉式） | 2 |
| M4 命名路由（@Noel → agent） | 1 |
| M13 Daemon 基础 runner | 1-2 |
| **小计** | **7-10 人月** |

→ 自然月：**2-3 个月**（4 人并行，M1 是关键路径）
→ 产出：一个能让人和多个 named agent 在共享频道里 @ 对方、agent 有持久记忆的 demo。**这是"开源替代 Raft"的最小可用形态，但还没有真正的 C3 协调工程。**

### Phase 1 — C3 协调层：Raft 真正值钱的部分

| 模块 | 人月 |
|------|------|
| M5 Held draft 新鲜度协议 | 2-3 |
| M6 并发原语（lock/CAS/watch/presence） | 3-5 |
| M7 事件流房间模型 | 3-5 |
| M4+ 能力寻址 + 反硬化画像 | 2-3 |
| **小计** | **10-16 人月** |

→ 自然月：**3-5 个月**（M6/M7 串行性强，难以纯并行——分布式正确性要反复验证）
→ 产出：agent 之间不再"撞车"，计数游戏式 demo 能稳定跑到 20+，held draft 在房间移动时正确持留。**这一步做完，才算真正"复制了 Raft 的护城河"。**
→ **风险最高的阶段**：M7 要把 turn-based 快照房间改成事件流，是架构层重写，容易超时。

### Phase 2 — 差异化层：Raft 没做的（超越的关键）

| 模块 | 人月 |
|------|------|
| M8 记忆可检视治理（browser/diff/provenance/branch/ACL） | 4-6 |
| M9 分级自治 trust tier + 可视化 | 2-3 |
| M10 多视图 Work Graph（chat/board/graph/diff/timeline） | 4-6 |
| **小计** | **10-15 人月** |

→ 自然月：**3-5 个月**（M8/M10 可并行）
→ 产出：功能上**开始超过 Raft**——记忆可审计、agent 有信任边界、工作有多视图。**这一步是"超越"vs"复制"的分水岭。**

### Phase 3 — 企业 + 生态：拉开身位

| 模块 | 人月 |
|------|------|
| M11 Enterprise（SSO/RBAC/审计/air-gap，借 LibreChat） | 3-5 |
| M12 AX 开放协议 + SDK | 3-5 |
| M14 OpenHands SWE vertical 接入 | 2-3 |
| **小计** | **8-13 人月** |

→ 自然月：**3-4 个月**
→ 产出：企业敢用 + 第三方 agent 可接入 + SWE 深度借力 OpenHands。**完全超越 Raft 的形态。**

### 总计

| 阶段 | 累计人月 | 累计自然月 | 里程碑 |
|------|---------|-----------|--------|
| Phase 0 | 7-10 | 2-3 | PoC，能 demo |
| + Phase 1 | 17-26 | 5-8 | 复制 Raft 护城河（C3 完成） |
| + Phase 2 | 27-41 | 8-13 | 功能上超过 Raft |
| + Phase 3 | 35-54 | 11-17 | 企业+生态，完全超越 |

**换算到不同团队规模**（自然月，含 Phase 0-3）：
- 3 人：**~18-24 个月**（关键路径串行受限）
- 5 人：**~12-17 个月**（较优配置）
- 8-10 人：**~10-14 个月**（M6/M7 仍串行，加人边际递减）
- 12+ 人：**~9-12 个月**（接近加人上限，协调成本反噬）

> 加人不能无限缩短：M6（并发原语）和 M7（事件流房间）是**正确性串行**任务，加人只会增加沟通成本，不会更快——这是 Brooks 定律的典型场景。

## 三、不同口径的"完成"对应的时间

| 目标 | 时间（5 人基线） | 含义 |
|------|----------------|------|
| 开源 PoC（证明能做） | 3-4 个月 | Phase 0，能 demo，无真正 C3 |
| 自托管可用版（小团队真用） | 6-9 个月 | Phase 0+1，追平 Raft 底座 + 基础协调 |
| 功能超越 Raft | 12-18 个月 | + Phase 2 差异化层 |
| 完全超越（企业+生态） | 18-24 个月 | + Phase 3 |

## 四、关键路径与最大风险

**关键路径**：M7（事件流房间）→ M6（并发原语）→ M8（记忆治理）→ M12（AX 协议）。这四个串行，决定最短工期。

**三大风险**：

1. **M6/M7 分布式正确性（最大风险）**。锁、CAS、事件一致性在多 agent 并发写共享状态时极易出 race condition。这类 bug 难复现、难调试，实际工期可能比估算多 30-50%。Raft 自己也只做到"AX 缓解、未解决"——说明这层真的硬。
2. **M12 AX 协议设计不确定性**。AX 是"没有标准答案"的设计问题（option-space 怎么 surfacing、感知 hook 怎么定义），迭代轮次不可预测。建议先内部实现（Phase 1 的 M5/M6 里隐含），稳定后再开放（Phase 3），避免协议早固化。
3. **LLM 成本/延迟工程**。Raft 用户峰值 1.2B tokens/day——一旦你的 PoC 跑起来真用，token 成本和 p99 延迟会逼出独立的工程问题（缓存、路由、批处理），这没算进上面的人月，可能额外 +2-4 人月。

**两个没算进去的隐藏成本**：
- **缝合成品的"接缝工程"**：LibreChat 的 agent 模型 ↔ Letta 的 memory_blocks ↔ 你的 C3 路由，三套抽象缝合的接缝（身份映射、状态同步、一致性问题）容易被低估，实际可能吃掉 +3-5 人月。这是 [06] 说"组合替代 ~70-80%"那缺失 20-30% 的具体出处。
- **产品化打磨**：onboarding、错误提示、可观测、计费、文档——从"能跑"到"别人能用"通常 +20-30% 工期。

## 五、复用开源到底省了多少

| 若全自研（不复用） | vs 复用开源 |
|------------------|-----------|
| Phase 0：~18-25 人月（自造 chat UI + 自造记忆引擎 + daemon） | 7-10 人月 → **省 ~50-60%** |
| Phase 1-3：基本不省（C3/差异化/协议开源无供给） | 持平 |
| 总计：~60-90 人月 | ~35-54 人月 → **省 ~35-45%** |

**结论**：复用开源主要省在 Phase 0（外壳 + 记忆底座），C3 及之后的差异化层开源帮不上。**最值钱的部分省不了**——这呼应 [06] 的核心判断。

## 六、对决策的建议

1. **若目标是"证明开源能替代 Raft"**：做 Phase 0（3-4 个月）即可出 PoC，但**它不会有 Raft 的护城河**（缺 C3），会被 Raft 的协调体验拉开差距。
2. **若目标是"做一个能用的自托管替代品"**：做到 Phase 1（6-9 个月），追平 Raft 底座 + 基础协调，能服务小团队。
3. **若目标是"超越 Raft"**：必须做到 Phase 2+（12-18 个月），否则只是另一个开源 chat+agent 项目，没有差异化。
4. **若资源有限、想最快出价值**：**跳过自造 C3，直接做 AX 协议层 + 接 OpenHands**（[04] 战略 6+7）——做"协议/协作层"而非"又一个 workspace"，让开源 vertical agent 接入你。这条路 Phase 0 + M12 + M14 ≈ 6-8 个月能出第一个生态位，且避开 M6/M7 的最难正确性工程（协议层把协调问题下放给接入方）。

---

## 一句话

> 开源欠缺的部分（C3 协调 + 缝合成品 + 差异化层），**3-5 人团队复用开源，追平 Raft 底座需 6-9 个月，功能超越需 12-18 个月，完全超越（企业+生态）需 18-24 个月**。其中 M6 并发原语 + M7 事件流房间是最难、最可能超时的关键路径；复用开源主要省 Phase 0 的外壳和记忆底座，最值钱的 C3 协调层省不了——这与 [06] "开源给零件、给不了灵魂"的判断一致。
