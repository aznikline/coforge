# 08 · 绕开 C3 泥潭：哪些 feature 更能成功

> 问题：M6（并发原语）+ M7（事件流房间）是最难、最可能超时、且 Raft 自己都没真正解决的部分。**不做这部分，有没有别的 feature 路线更能成功？**
>
> **核心判断**：有。而且绕开 C3 反而可能**比硬刚更成功**。理由是 Raft 的成功学——它赢在"找被忽视的维度占位"，不在"做最难的事"。复制它的护城河（C3）是低 ROI 的苦活；**找到下一个被忽视的维度**才是高 ROI 的赢法。下文给出 7 条绕路 feature 路线，按"成功概率 × 可行性"排序。

---

## 一、为什么绕开 C3 反而更可能成功

先把"不做 C3"的代价和收益摆清楚：

**代价**：放弃"在多 agent 实时协调正确性上超过 Raft"这个点。Raft 的计数游戏式 demo 你做不出同等水平的。

**收益**（被低估的部分）：
1. **C3 是正确性泥潭**。分布式并发 bug 难复现、难调试，Raft 自己都只做到"AX 缓解、未解决"。你投入 6-10 人月可能只换来"和 Raft 打平还略逊"——这是低差异化、高风险的投入。
2. **用户其实不太感知 C3**。Raft 真正让用户买单的是"持久记忆 + 队友感 + 聊天界面"，不是"计数到 20 不冲突"。C3 是 Raft 的工程内功，不是用户购买理由。**你不需要在内功上赢，你需要在用户感知的维度上赢。**
3. **省下的 6-10 人月可以投到 Raft 完全没碰、且用户能直接感知的维度**——这才是差异化的来源。

**结论**：C3 做到"够用不撞车"即可（用轻量锁 + 串行化简化，2-3 人月），**不要追求"原生并发正确性超越 Raft"**。把省下的预算投到下面的绕路 feature。

## 二、7 条绕路 feature 路线（按成功概率排序）

### 路线 A：可迁移的"组织记忆"层 — 把记忆做成跨产品资产 ★★★★★

**是什么**：不做一个 workspace，而是做一层**可迁移的、属于用户的 agent 记忆层**——你的 agent 记忆能跟人走，接入任何 agent 平台（Raft / Cursor / Devin / Claude Code）。

**为什么更能成功**：
- 这是 Raft 记忆的**反向解构**。Raft 的记忆锁死在它的 workspace 里（vendor lock-in）——这正是它的软肋（[03] gap 3）。你做"记忆的可移植性"，直接把 Raft 的留存引擎变成你的获客引擎。
- **叙事占位**：Raft 占了"agent 队友"；你占"**你的记忆归你**"（Your memory, your asset）。这是一个 Raft 无法反驳的范式词——它不能说"记忆不该可迁移"。
- **切的是 LLM 时代真痛点**：用户最大的恐惧是"被某个 AI 平台锁死"。一个"记忆护照"产品天然有需求。
- **C3 几乎不需要**：记忆层是单 agent 的存储 + 检索 + 治理，没有多 agent 实时协调问题。M8（记忆可检视治理）是这个路线的核心，而 M8 难度是中-高，远低于 M6/M7。

**工程量**：~8-12 人月（M8 为主 + 各平台 adapter + SDK）。**无 C3 依赖**。
**风险**：各平台没有标准记忆接口（MCP 可借力——见路线 G）。需要自己定义"记忆协议"。

### 路线 B：人机分工编译器 — 从"协作"升到"分工规划" ★★★★★

**是什么**：不做"人机共享频道协作"，做一层**分工规划器**——给定一个目标，系统把工作拆成"人做哪些 / agent 做哪些 / 谁先谁后 / 哪些需要人审批"，输出一个可执行的分工 DAG，然后调度。

**为什么更能成功**：
- **升一个抽象层**。Raft 在"执行层协作"（agent 在频道里干活）卷；你在"规划层分工"（谁来干、怎么串）卷。规划层比执行层更接近"决策价值"，用户愿意为决策付费多过为执行付费。
- **绕开实时协调**：分工 DAG 是**离线规划 + 调度**，不需要多 agent 实时并发协调——天然避开 M6/M7。agent 按调度顺序执行，冲突在规划阶段就消解了。
- **叙事占位**：Raft = "队友"；你 = "**人机分工的编译器**"（compile intent into a human+agent execution plan）。这个隐喻比"队友"更精准地抓住了真实痛点——人不知道什么该交给 AI。
- **可观测性强**：分工 DAG 天然是一个可视化对象（图），比 chat 流更易做"可审计"。

**工程量**：~10-14 人月（规划 LLM + DAG 引擎 + 调度器 + 可视化）。**轻 C3**（调度是串行/受控并发，非实时）。
**风险**：规划质量依赖 LLM 推理——需要好的分解 prompt + 验证循环。但这比分布式正确性好做得多。

### 路线 C：Agent 可观测性 + 行为回放 — 做 agent 的 Datadog ★★★★

**是什么**：不做 agent runtime，做**agent 行为的观测层**——trace 每个决策、token、工具调用、记忆读写，可回放、可 diff、可设告警。支持接入 Raft / 自建 agent / 任何 runtime。

**为什么更能成功**：
- **完全绕开 C3**：观测是只读旁路，不参与协调，零并发正确性问题。
- **切企业真痛点**：企业不敢用 agent 的头号原因不是"能力不够"，是"看不见它干了什么、为什么"。可观测性是企业采购的硬门槛，Raft 这块极弱（[03] gap 3、5、6）。
- **叙事占位**：Raft = "队友"；你 = "**agent 的黑匣子记录仪**"。一个 trust-native 的叙事，恰好补 Raft "throw them into the work"的安全漏洞。
- **商业化清晰**：观测类产品（Datadog 模式）有成熟的可复制计价（按 trace/ingest 量）。
- **生态卡位**：做观测天然要接所有 runtime——你变成**跨平台的 agent 治理层**，比绑定自己 workspace 的 Raft 生态位更宽。

**工程量**：~8-12 人月（trace 采集 SDK + 存储 + 回放 UI + 告警）。**零 C3**。
**风险**：需要各 runtime 暴露 hook（MCP/OpenTelemetry 可借力）。先支持 1-2 个（OpenHands、自研）再扩。

### 路线 D：垂直纵深 — 只做一个垂直，但做到 Raft 做不到的深 ★★★★

**是什么**：放弃"通用协作空间"，**只挑一个垂直**（如 SWE，或法律/数据/设计），做一个比 Raft 深、比 Devin 协同的垂直产品。

**为什么更能成功**：
- **避开 Raft 的通用主场**。Raft 的 0.1 席 + 聊天界面是横向通用的护城河；你在横向打不过。但 Raft 每个垂直都浅（[03] gap 7）——在单垂直纵深上，Raft 不是对手。
- **C3 需求降低**：单垂直、单 agent 类型为主时，多 agent 实时协调的需求大幅下降。比如 SWE 里一个长 horizon agent + 受控的 sub-agent，不需要"20 个 agent 在频道里协调"。
- **叙事占位**：Raft = "通用队友"；你 = "**某垂直的深度专家团队**"。深度在特定人群里比通用更有购买力。
- **借力开源**：SWE 垂直可直接 fork OpenHands（MIT）做底座，省 M2/M4 大量工作。

**工程量**：~10-15 人月（垂直深度 + 受控协作）。**轻 C3**。
**风险**：选错垂直代价大。SWE 最卷（Devin/Cursor）；可考虑**数据工程、DevOps、合规审计**等 Raft/Devin 都没深耕的垂直。

### 路线 E：Agent 经济 / 市场 — 做交易层而非协作层 ★★★

**是什么**：不做协作 workspace，做**agent 服务的市场 + 计费 + 信任评级**——agent 提供方挂牌、消费方按用量付费、平台做质量评级与担保。

**为什么更能成功**：
- **完全不同的赛道**。Raft 是 SaaS 协作工具；你是 agent 时代的 marketplace。不在同一战场，无直接竞争。
- **网络效应**。市场有双边网络效应，一旦运转，护城河比任何 feature 都深——这是 Raft 没有的结构性优势。
- **叙事占位**：Raft = "你的队友"；你 = "**agent 劳动力市场**"。
- **绕开 C3**：市场是交易撮合 + 评级，不涉及实时多 agent 协调。

**工程量**：~12-16 人月（挂牌/计费/评级/担保/纠纷）。**零 C3**。
**风险**：双边市场冷启动难（先有鸡还是先有蛋）。需要先在一个垂直攒起供给。这是**商业模式风险**，不是工程风险。

### 路线 F：本地优先 / 离线 agent 工作站 — 隐私极致 ★★★

**是什么**：押注"**所有 agent 数据与执行 100% 本地、可完全离线**"——比 Raft 的自托管 daemon 更极端：本地模型 + 本地记忆 + 本地协调，air-gap 友好。

**为什么更能成功**：
- Raft 虽自托管但仍依赖云 LLM API（隐私没到顶）。**全本地**是一个 Raft 结构上难以追上的位置——它的持久记忆/协调架构是按云 LLM 设计的。
- **切合规敏感客户**：金融/医疗/政府/律所——这些 Raft 的 Enterprise "Coming soon" 还够不到（[03] gap 6）。
- **叙事占位**：Raft = "自托管队友"；你 = "**离线优先的 agent 工作站**"。
- **绕开 C3**：单机/本地小团队的 agent 协调，并发规模小，M6/M7 难度骤降（单机锁即可）。

**工程量**：~10-14 人月（本地模型集成 + 本地记忆 + 单机协调）。**轻 C3**（单机）。
**风险**：本地模型能力上限低于云模型——产品力受限于本地 LLM 进展。但这个"限制"恰好是它的卖点（隐私换能力，特定客户愿接受）。

### 路线 G：MCP-for-Agents — 做协议层而非产品层 ★★★★

**是什么**：不做 workspace，做**agent 间协作的开放协议 + 参考实现**——类似 MCP 之于工具调用，你做"agent 间寻址/委派/交接"的协议标准。

**为什么更能成功**：
- **协议拥有者 > 产品拥有者**（[04] 战略 6 已论证）。Raft 是私有产品；你是开放协议。一旦成为事实标准，所有 agent 平台都得接你——生态位远高于任何单一产品。
- **叙事占位**：Raft = "产品"；你 = "**agent 互联的协议**"。这是品类定义权，比 Raft 的 DAA 度量更上游。
- **绕开 C3 的正确性部分**：协议层只定义"agent 间如何寻址/委派/交接的消息格式"，**把协调的正确性下放给各实现方**——你不背多 agent 并发 bug 的锅，各 runtime 自己负责。这是最聪明的绕路：定义规则，不亲自动手跑规则。
- **借力趋势**：MCP 已经被 Anthropic 推开，"协议化"是行业共识。做"agent 间协议"踩在风口上。

**工程量**：~6-10 人月（协议设计 + 参考实现 SDK + 2-3 个 demo runtime 适配）。**零 C3**（下放给实现方）。
**风险**：协议采纳不确定性高——需要早期就有 1-2 个知名 runtime 背书。可先以开源参考实现 + 接 OpenHands 证明可用。

## 三、路线对比矩阵

| 路线 | 成功概率 | 可行性(工程量) | C3 依赖 | 叙事占位强度 | 网络效应 | 商业模式清晰度 |
|------|:------:|:------:|:------:|:------:|:------:|:------:|
| A 可迁移记忆层 | 高 | 高(8-12pm) | 无 | 强 | 中 | 高(SDK+订阅) |
| B 人机分工编译器 | 高 | 中(10-14pm) | 轻 | 强 | 低 | 高(按规划/执行) |
| C agent 观测/回放 | 高 | 高(8-12pm) | 无 | 中-强 | 中(跨平台) | 高(ingest 计费) |
| D 垂直纵深 | 中-高 | 中(10-15pm) | 轻 | 中 | 低 | 高(垂直 SaaS) |
| E agent 市场 | 中 | 中(12-16pm) | 无 | 强 | **高** | 中(抽佣) |
| F 本地优先工作站 | 中 | 中(10-14pm) | 轻(单机) | 中 | 低 | 中(许可/订阅) |
| G agent 间协议 | 中-高 | 高(6-10pm) | 无 | **强** | 高(生态) | 低-中(协议免费,靠实现/托管) |

> pm = 人月。工程量越低 + C3 依赖越低 = 越能快速出价值且避开泥潭。

## 四、推荐组合：绕开 C3 的最优策略

不建议单押一条，推荐**两两组合**形成更有壁垒的产品：

### 组合 1（最高成功概率）：A + C = "可迁移记忆 + 可观测"
- 记忆层 + 观测层 = **trust-native agent infrastructure**。
- 叙事："你的 agent 记忆归你，且你能看见它做的一切。"
- 完全无 C3，工程量 ~14-20 人月（5 人 ~4-5 个月出可用版），切的是企业最硬的两个门槛（记忆治理 + 行为审计）——Raft 全空白。
- 这是最像"Raft 没做的工程打包成产品"的路线，差异化清晰且不撞 C3。

### 组合 2（最高叙事势能）：B + G = "分工编译器 + 开放协议"
- 规划层做产品，协议层做生态。
- 叙事："把人机分工编译成执行计划，并用开放协议让任何 agent 执行。"
- 工程量 ~14-20 人月，C3 下放给协议实现方。一旦协议被采纳，生态位高于 Raft。

### 组合 3（最快出价值）：G + 单一 vertical（如 D 的 SWE fork OpenHands）
- 先做协议 + 一个 demo vertical，证明协议可用。
- 工程量 ~10-14 人月，6-8 个月出生态位（[07] 的捷径路线）。

## 五、对原战略 [04] 的修正

[04] 的战略 2（原生并发原语）和战略 6（AX 协议）原来放在一起。本文的结论是：**应该把战略 2 降级、战略 6 升级**：

- **战略 2（C3 并发原语）降级为"够用即可"**：只做轻量锁 + 串行化简化（2-3 人月），不追求超越 Raft 的协调正确性。承认这是 Raft 的主场，不硬刚。
- **战略 6（AX/agent 间协议）升级为核心**：从"做成开放协议"升级为"**产品就是协议层**"（路线 G）。把协调正确性下放给实现方，自己只占协议 + 生态位。
- **新增差异化重心**：把省下的预算投到路线 A（可迁移记忆）或 C（观测）——这是 Raft 全空白、用户能直接感知、且无 C3 依赖的方向。

## 六、最终判断

> **能不做 C3 吗？能。而且绕开它做别的，很可能比硬刚更成功。**
>
> Raft 的成功学不是"做了最难的事"，是"找了被忽视的维度占位"。复制它的 C3 内功是低 ROI 苦活（6-10 人月换"打平还略逊"）；**找下一个被忽视的维度**才是高 ROI 赢法。
>
> 最优解：C3 做到"够用不撞车"（2-3 人月轻量版），把主力投到 **A 可迁移记忆层 + C agent 观测**（无 C3、切企业硬门槛、Raft 全空白）或 **G agent 间协议 + B 分工编译器**（协议生态位、正确性下放）。这两条路都比"在 C3 上超越 Raft"更可能成功——因为它们要么占了 Raft 没占的范式词，要么切了 Raft 没切的客户，要么建了 Raft 没有的网络效应，且都绕开了最难最易超时的分布式正确性泥潭。
